# Brief: 3.2b A-MEM — PPR-hybrid duplicate finding

**Target:** neuron-hq
**Effort:** 1 körning
**Roadmap:** Fas 3 — Agent-mognad, punkt 3.2 (del 2 av 2)

## Bakgrund

Brief 3.2a flyttade Consolidator till orchestratorn och lade till abstraktionsverktyg. Denna brief adresserar den tredje begränsningen:

**Jaccard + Embeddings för kandidathittning.** `findDuplicateCandidates()` jämför titlar med Jaccard-similarity och kör semantisk sökning via embeddings som fallback (om tillgängliga). Noder med helt olika titlar men samma underliggande koncept hittas dock inte tillförlitligt — embeddings fångar semantisk likhet men inte strukturella kopplingar i grafen. PPR (HippoRAG) finns i `src/core/ppr.ts` som `personalizedPageRank()` men används inte av Consolidator.

Denna brief lägger till:
1. PPR-baserad kandidathittning (nytt tool)
2. Hybrid-scoring i befintlig duplicate finding (Jaccard + PPR-boost)
3. PPR Discovery-sektion i Consolidator-prompten
4. Utökad findings-rapport med PPR-upptäckter och grafstatistik

## Designbeslut

### 1. Nytt verktyg: find_ppr_candidates

| Verktyg | Input | Output | Vad det gör |
|---------|-------|--------|-------------|
| `find_ppr_candidates` | `nodeId`, `limit?` | Array av `{node, score}` | Kör PPR från en given nod och returnerar topp-N kandidater. Filtret: exkluderar noder som redan har direkt kant till seed. Ger Consolidator möjlighet att hitta semantiskt relaterade noder som Jaccard missar. |

### 2. PPR-integration i duplicate finding

`find_duplicate_candidates` utökas med en hybrid-approach:
1. Jaccard (befintlig, snabb) — hittar textuellt liknande noder
2. Embeddings (befintlig fallback) — semantisk likhet om tillgängliga
3. PPR-boost (NY) — för varje Jaccard-kandidatpar från steg 1, kör PPR från nod A: om nod B har hög PPR-score, höj similarity-poängen

**OBS: PPR-boost körs ENBART på Jaccard-kandidater (steg 1).** Embeddings-kandidater (steg 2) tillkommer i handleren (`executeFindDuplicateCandidates`) EFTER att `findDuplicateCandidates()` returnerat — de tre signalerna är separata: Jaccard+PPR i `graph-merge.ts`, embeddings i handleren. Ingen embeddings-kandidat PPR-boostas.

**Formel:** `final_score = jaccard * 0.6 + ppr_proximity * 0.4`

Där `ppr_proximity = ppr_score(B | seed=A)` normaliserat till [0, 1].

**Normaliseringsmetod:** För varje seed-nod, kör PPR och samla alla kandidat-scores. Normalisera genom att dela varje score med `max(ppr_scores)` för det seedet (min-max med min=0). Om `max(ppr_scores) < 1e-6` (epsilon-tröskelvärde) sätts `ppr_proximity = 0` för alla — förhindrar att floating-point-brus ger falska positiver. Detta ger relativ ranking inom varje seed, inte absoluta PPR-värden.

**Batch-begränsning:** Kör PPR-boost för max 50 kandidatpar per körning. Om Jaccard producerar fler än 50 kandidater, PPR-boosta de 50 med högst Jaccard-score och behåll resten med enbart Jaccard-score. Detta förhindrar att N×PPR-körningar blir en flaskhals på stora grafer.

### 3. PPR Discovery i prompten

Consolidator-prompten får en ny sektion "PPR-driven Discovery" (efter "Merge Duplicates"):
- Använd `find_ppr_candidates` för att hitta dolda kopplingar
- PPR hittar noder som Jaccard missar — använd det
- OBS: PPR i hybrid duplicate finding (`findDuplicateCandidates` med `usePpr`) är en *boost* — inte en självständig duplikat-signal. `find_ppr_candidates` är verktyget för ren PPR-discovery
- Consolidator-toolhandlern (`executeFindDuplicateCandidates` i consolidator.ts) anropar `findDuplicateCandidates(graph, threshold, { usePpr: true })` — PPR-boost aktiveras i handleren. OBS: handleren kör redan embeddings-fallback (sök efter `isEmbeddingAvailable` i handleren) — PPR-boost sker i `graph-merge.ts` FÖRE handleren lägger till embeddings-resultat. (Radnummer verifierade av Code Anchor S136 R2 — agenten bör ändå söka strukturellt som fallback)

### 4. Priority Order-uppdatering

export function findPprCandidates(
  graph: KnowledgeGraph,
  nodeId: string,
  options?: { limit?: number; excludeDirectNeighbors?: boolean }
): Array<{ node: KGNode; score: number }> {
  // 1. Adaptera KnowledgeGraph till personalizedPageRank()-format:
  //    - nodes: graph.nodes.map(n => n.id)
  //    - edges: graph.edges.map(e => ({ from: e.from, to: e.to }))
  //    - seeds: new Map([[nodeId, 1.0]])
  // 2. Kör personalizedPageRank(nodes, edges, seeds)
  // 3. Exkludera direkta grannar (om flagga satt, default true)

**Import:** `findPprCandidates()` i `graph-merge.ts` importerar `personalizedPageRank` och `PPRResult`-typen som named imports från `src/core/ppr.ts`. `PPRResult` är exporterad som `export interface PPRResult { nodeId: string; score: number }`.

## Risker

| Risk | Sannolikhet | Konsekvens | Mitigation |
|------|-------------|------------|------------|
| PPR-hybrid gör duplicate finding långsammare | Låg | Längre körtid | PPR är O(edges × iterations), <1s på 1000 noder. Fallback: `usePpr: false` |
| `personalizedPageRank()`-adaptern i `findPprCandidates()` kan mappa fel | Låg | Felaktiga PPR-scores | Signaturen är verifierad: `personalizedPageRank(nodes: string[], edges: Array<{from, to}>, seeds: Map<string, number>, options?)` returnerar `PPRResult[]` med `{nodeId, score}`. Adapter-logiken är enkel — skriv tester tidigt |
| Batch-begränsning (50 par) är för liten/stor | Låg | Missade kandidater / långsam körning | Konfigurerbar via options — kan justeras i framtiden |

## Acceptanskriterier

### PPR-integration

- **AC1:** `findPprCandidates()` returnerar noder rankade av PPR-score, exkluderar direkta grannar. Kastar `Error` om `nodeId` inte finns i grafen (följer `mergeNodes()`-konventionen i `graph-merge.ts` och `personalizedPageRank()`s throw vid ogiltigt seed). Returnerar tom array om grafen har < 2 noder eller om seed-noden saknar kanter (isolerad nod — ingen PPR-traversal möjlig)
- **AC2:** `findDuplicateCandidates()` stödjer `usePpr`-option med hybrid-scoring (jaccard × 0.6 + ppr × 0.4). Om grafen har < 2 noder med `usePpr: true` returneras tom array (ingen fallback behövs — Jaccard producerar heller inga kandidatpar). Om `personalizedPageRank()` kastar exception under PPR-boost, falla tillbaka på ren Jaccard-score för det berörda paret och logga varning
- **AC3:** `findDuplicateCandidates()` med `usePpr: true` kör PPR-boost för max 50 kandidatpar; överskjutande kandidater behåller enbart Jaccard-score
- **AC4:** Utan `usePpr` beter sig `findDuplicateCandidates()` exakt som idag — befintliga anropsställen i codebasen kompilerar utan ändringar

### Prompt

- **AC5:** Consolidator-prompten har sektion "PPR-driven Discovery"
- **AC6:** "Strengthen connections"-posten i Priority Order uppdaterad med PPR-tillägg (oavsett nuvarande position)

### Tester

Inkludera: (a) ett explicit epsilon-guard-testfall: när alla PPR-scores < 1e-6 ska `ppr_proximity = 0` returneras (inte NaN eller Infinity), (b) ett batch-gränsvärdestest: 51 Jaccard-kandidater → de 50 med högst Jaccard-score PPR-boostas, den 51:e behåller sin rena Jaccard-score
- **AC9:** Alla befintliga tester passerar utan regression
